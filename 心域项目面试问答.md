# 心域项目面试问答准备

## 一、RAG 双路召回架构

### Q1：你的 RAG 架构是怎么设计的，为什么用双路召回？

我用策略模式 + 工厂模式设计了 RAG 架构。核心入口是 `UnifiedRagServiceImpl`，通过 `RagStrategyFactory` 管理多个检索策略，包括 `LocalMemoryStrategy`（本地关键词匹配）、`BailianKnowledgeStrategy`（阿里云百炼知识库）和 `HybridRagStrategy`（混合策略）。

双路召回的原因是单一数据源覆盖不全：本地 ES 擅长结构化知识检索（BM25 + 向量混合），但知识量有限；百炼知识库有更丰富的外部心理知识，但延迟相对高。两路并行检索后，`HybridRagStrategy` 用 `CompletableFuture` 并行调用，10 秒超时，然后合并去重（取内容前 100 字符做 key 去重）、加权排序（百炼权重 1.2，本地权重 1.0），最后按综合评分选最佳结果送给 LLM。

### Q2：ES 的混合搜索具体怎么实现的？

在 `ElasticsearchRagServiceImpl` 里，我用 `NativeQuery` 构建了一个 bool 查询 + KNN 向量搜索的组合：

- bool 查询里有两个 should 子句：`content` 字段 match（boost 1.0）和 `title` 字段 match（boost 2.0）
- 同时叠加 `KnnSearch`，对 `embedding` 字段做 K 近邻搜索，候选数设为 topK 的 10 倍

两路结果由 ES 内部的 `max_score` 融合。`embedding` 是 1024 维 `dense_vector`，similarity 用 cosine，通过阿里云百炼 `text-embedding-v3` 模型生成。

### Q3：ES 挂了怎么办？

`RagKnowledgeServiceImpl.retrieve()` 里有降级策略：优先调用 ES 混合搜索，如果 ES 未初始化（`isInitialized()` 返回 false）或搜索抛异常，catch 住后回退到 MySQL 全文检索，用 `LIKE` 模糊匹配 `title`、`content`、`summary` 字段，再按关键词出现频率计算一个简单的相关性评分排序。

### Q4：为什么选 cosine 相似度而不是 dot product？

因为向量是经过归一化的（模型输出），cosine 和 dot product 在这种情况下等价，但 cosine 语义更明确，而且 ES 的 `dense_vector` 对 cosine 的实现更稳定。如果用 dot product，当向量长度不一致时会导致分数偏移。

### Q5：什么是 RAG？RAG 的完整流程是什么？

RAG（Retrieval-Augmented Generation）是检索增强生成，核心思想是先从知识库检索相关内容，再把检索结果作为上下文塞进 Prompt 给 LLM 生成回答。

我项目里的流程是：用户输入 Query → 模板方法定义的标准流程（Query 改写 → 向量召回/知识库检索 → 重排序合并 → 拼接上下文 + Prompt → LLM 生成）。比纯 LLM 对话的好处是回答有知识依据，不容易幻觉，尤其在心理专业领域很关键。

## 二、多人格对话 + Prompt 工程

### Q6：你封装了哪几种人格，怎么切换的？

三种人格：

- **listener（倾听者/温柔）**：先共情再回应，150 字以内，多用"我感受到"、"谢谢你愿意告诉我"
- **analyst（分析者/理性）**：朋友式梳理情绪，引导换角度看问题，120 字以内
- **healer（疗愈者/活泼）**：鼓励型，强调力量和希望，200 字以内

人格用 `ConcurrentHashMap<String, String>` 维护用户-人格映射。优先级：请求参数 > 用户已设置 > 默认 listener。切换通过 `POST /api/v1/chat/persona` 接口。

### Q7：模板方法模式在这里怎么体现？

对话生成的流程是固定的：获取人格 → 构建历史上下文（最近 10 条）→ 获取 system prompt → 拼接完整 prompt → 调用 LLM 生成。其中"获取哪种人格的 system prompt"是变化点，三种人格有不同的 prompt 模板。流程骨架固定，具体 prompt 可替换，这就是模板方法模式的思想。

### Q8：为什么限制字数？

心理对话场景下，青少年用户需要的不是长篇大论，而是温暖的、有针对性的短回应。字数太长反而让用户觉得冰冷、模板化。不同人格的字数限制不同，温柔型需要更多表达所以 150 字，理性型强调简洁所以 120 字，疗愈型需要情感铺垫所以 200 字。

## 三、SSE 流式对话

### Q9：SSE 流式输出怎么实现的，首字延迟怎么控制？

后端用 `SseEmitter` 创建 SSE 连接，通过 `CompletableFuture.supplyAsync` 异步处理，不阻塞 Tomcat 线程。LLM 调用用 LangChain4j 的 `StreamingChatModel`，`onPartialResponse` 回调逐 token 推送，每个 token 通过 `emitter.send(SseEmitter.event().name("chunk").data(...))` 推给前端。

SSE 事件协议：`start`（连接建立）→ `chunk`（内容片段）→ `done`（完成）→ `error`（异常）。

首字延迟控制：后端 LLM 首个 token 返回后立即推送，不走完整生成再返回。网络层面 SSE 是长连接，无需重复建立连接。实测首字延迟在 200ms 以内。

### Q10：观察者模式怎么体现的？

两层观察者模式。后端 `MindRealmAdvisor` 实现了 Spring AI 的 `CallAdvisor` 和 `StreamAdvisor`，在 LLM 调用前后做日志观察（记录请求、响应、耗时）。前端用观察者模式封装 SSE 订阅，当收到 chunk 事件时通知所有订阅者逐字渲染到页面，解耦了数据接收和 UI 渲染。

### Q11：怎么防止用户重复提交流式请求？

用 `ConcurrentHashMap<Long, AtomicBoolean> activeRequests` 做 CAS 锁。每个用户发流式请求时 `compareAndSet(false, true)`，如果已有一个在处理就直接拒绝返回错误。请求完成或异常时 `remove` 释放锁。

## 四、Kafka 异步架构

### Q12：为什么引入 Kafka，不直接同步调用？

三个痛点：1）ES 同步写入如果 ES 挂了或慢，会拖慢用户写日记的响应甚至失败；2）风险预警分析涉及关键词匹配和可能的 LLM 语义分析，同步调用增加用户等待；3）短信/邮件是第三方服务，不稳定。

Kafka 解耦后，主业务操作只做 DB 写入 + 发消息，消费端独立处理，互不影响。

### Q13：AOP 切面是怎么发布事件的？

自定义 `@EsSync` 注解，标记在 Diary 和 Story 的增删改方法上。`EsSyncAspect` 用 `@AfterReturning` 拦截方法成功返回后的结果：

- SAVE 操作：从返回值反射取 `id` 字段（支持 Long 直接返回、实体对象反射、Boolean 返回时从参数取 id）
- DELETE 操作：从方法参数取第一个 Long 类型参数

然后构建 `EsSyncEvent`（entityType + operation + entityId），按 `entityType` 作为 key 发到 Kafka，保证同类型消息路由到同一分区，消费有序。

目前有 6 个方法加了 `@EsSync`：Diary 的 create/update/delete、Story 的 publishStory/auditStory/deleteStory。

### Q14：Kryo 序列化为什么比 JSON 好？

Kryo 是二进制序列化，体积小、速度快。JSON 序列化有字段名冗余，1024 维的 float 数组 JSON 化后非常大。实测 Kryo 序列化后体积约为 JSON 的 1/3，反序列化速度快 5-10 倍。

用了一个对象池（max 32 个 Kryo 实例），预注册了 `EventEnvelope`、`EsSyncEvent` 等核心类型，线程安全。`EventEnvelope` 是统一事件信封，包含 `eventId`（UUID 幂等）、`eventType`、`payload`（泛型）、`timestamp`、`source`、`retryCount`、`metadata`。

### Q15：Redis 幂等去重怎么做的？

每个消费者用 `SETNX`（`setIfAbsent`）做幂等：key 格式是 `kafka:dedup:es:{eventId}`（24h TTL）。消费前 `SETNX`，成功说明没处理过；消费失败时 `delete` 释放锁，允许重试。三个消费者分别用 `es:`、`warn:`、`notif:` 前缀隔离。

### Q16：消息消费失败了怎么办？

消费者 catch 异常后：1）释放 Redis 幂等锁（`delete key`）；2）抛 `RuntimeException` 触发 Spring Kafka 的 `DefaultErrorHandler`。`DefaultErrorHandler` 配了指数退避（1s → 2s → 4s → 8s），重试耗尽后转发到 `.DLT` 死信队列。`DeadLetterHandler` 统一监听所有 DLT Topic，只记日志不重试，预留了告警和管理端手动重放接口。

### Q17：Kafka Producer 的可靠性怎么保证？

`acks=all`（所有副本确认）、`enable.idempotence=true`（生产者幂等，防止网络重发导致重复）、`compression.type=lz4`（压缩减少网络开销）。Consumer 侧 `isolation.level=read_committed`，只读已提交的消息。

### Q18：三个 Topic 的分区和路由策略是什么？

三个 Topic 都是 3 分区 1 副本：

- `mind-realm.es.sync`：按 `entityType`（diary/story）路由，同类型消息有序
- `mind-realm.warning.analyze`：按 `userId` 哈希路由，同一用户的消息有序
- `mind-realm.notification.send`：按通知类型路由

消费者 `concurrency=3`，对应 3 个分区并行消费。

## 五、ES 同步模板方法模式

### Q19：BaseEsSyncService 怎么体现模板方法模式？

`BaseEsSyncService<E, D>` 是抽象基类，定义了 `fullSync` 和 `rebuildIndex` 的固定骨架：分页查 DB → 转换为 ES 文档 → 批量写入。其中 `fetchFromDb`、`convertToDocument`、`fetchByIdFromDb`、`countInDb` 四个抽象方法留给子类实现。

`DiaryEsSyncServiceImpl` 和 `StoryEsSyncServiceImpl` 分别继承了它，实现了自己的 DB 查询和实体转换逻辑。新增同步类型只需加一个子类，不用改公共逻辑。

同时它们还实现了 `EsSyncHandler` 接口，被 Kafka 消费者的 `handlerMap` 路由调用，一个类承担两个角色。

## 六、情感分析与风险预警

### Q20：风险预警的逻辑是什么？

`WarningServiceImpl.analyzeRisk()` 对内容做关键词匹配：

- 高风险（riskLevel=3）：自杀、轻生、不想活、suicide、kill myself
- 中风险（riskLevel=2）：绝望、无助、崩溃、抑郁、痛苦

riskLevel > 1 就插入 `warning_record` 表。riskLevel=3 且邮件通知开启时，调用 `notifyGuardian` 发通知。通知通过 Kafka `mind-realm.notification.send` Topic 异步处理，`NotificationConsumer` 根据 `notificationType` 决定渠道：warning/emergency 走短信+邮件，reminder 只写站内通知。

### Q21：为什么用 Kafka 发通知而不是直接调？

通知涉及第三方服务（阿里云短信、邮件 SMTP），延迟不确定。直接调的话如果短信服务超时，整个预警流程就卡住了。用 Kafka 解耦后，预警记录秒级写入 DB，通知异步发送，即使短信挂了也不影响预警记录的完整性。

## 七、设计模式汇总

### Q22：项目里用了哪些设计模式？

| 模式 | 场景 | 核心类 |
|---|---|---|
| 策略模式 | RAG 检索策略切换 | `RagStrategy` 接口 + `RagStrategyFactory` |
| 工厂模式 | RAG 策略管理、LLM Judge 模型切换 | `RagStrategyFactory`、Judge 工厂 |
| 模板方法 | ES 同步标准流程、对话生成流程 | `BaseEsSyncService<E,D>`、`AiChatServiceImpl` |
| 观察者模式 | SSE 消息推送、Spring AI Advisor | `MindRealmAdvisor`、前端 SSE 订阅 |
| AOP 切面 | ES 同步事件发布 | `@EsSync` + `EsSyncAspect` |
| 单例模式 | WebSocket 全局连接管理 | 全局单例管理器 |

### Q23：策略模式和模板方法模式有什么区别？

策略模式关注"做什么可以换"——不同策略实现同一个接口，运行时切换（比如 RAG 切换本地检索/百炼检索/混合检索）。

模板方法模式关注"流程固定，细节不同"——父类定义算法骨架，子类实现具体步骤（比如 ES 同步的"分页查DB → 转换 → 写入"流程固定，但不同实体的查询和转换逻辑不同）。

## 八、测试 Agent + LLM-as-Judge

### Q24：测试 Agent 是怎么工作的？

构建了一个基于 LLM 的测试 Agent，输入项目接口文档和需求描述，自动生成 50+ 条测试用例，覆盖正常流程、边界条件、异常情况。用例生成后可以一键执行，生成测试报告。

### Q25：LLM-as-Judge 是什么？

用 LLM 来评估 AI 回答的质量，代替人工打分。通过策略模式 + 工厂模式支持切换不同的 Judge 模型（比如 GPT-4、百炼等），每种模型有独立的评分策略。评估维度包括回答的相关性、准确性、安全性、共情程度等。

## 九、项目整体

### Q26：项目整体架构是什么样的？

```
前端：Uni-app 小程序 (Vue3) + React 管理后台 (Vite + Ant Design)
    ↓ HTTP/SSE
API 层：SpringBoot 3
    ├── Controller 层（接口、参数校验、RBAC）
    ├── AOP 层（ES同步切面、日志切面、限流切面）
    └── Service 层
        ├── RAG 模块（策略模式，双路召回）
        ├── AI 对话模块（Spring AI + LangChain4j）
        ├── 预警模块（关键词匹配 + Kafka 异步）
        └── 通知模块（Kafka → 站内信/短信/邮件）
基础设施：MySQL + Redis + ES + Kafka
部署：Docker Compose + Nginx + OpenClaw + Natapp 内网穿透
```

### Q27：这个项目最大的技术难点是什么？

RAG 双路召回的稳定性和延迟平衡。两路并行检索（ES 本地向量搜索 + 百炼知识库 HTTP 调用）的延迟差异大，需要合理设置超时（10 秒）和降级策略（一路失败不影响另一路）。合并去重和加权排序的逻辑也需要调试，避免重复知识片段影响 LLM 生成质量。

### Q28：你负责哪些部分？

全栈。后端从架构设计到编码：RAG 策略模式设计、Kafka 异步链路搭建、ES 混合搜索、AOP 切面、预警模块。前端：Uni-app 小程序开发（SSE 订阅 + WebSocket）、React 管理后台（RBAC + CRUD 泛型组件）。运维：Docker Compose 部署、OpenClaw 自动化、Natapp 内网穿透上线。

### Q29：如果让你重新做，会改什么？

1）风险预警目前只做关键词匹配，可以升级为 NLP 模型做语义理解，减少误报漏报；2）死信队列目前只记日志，应该写入 DB 表并做管理端查询和手动重放；3）Kafka 可以考虑加监控（Kafka Metrics 暴露到 Prometheus + Grafana）。
